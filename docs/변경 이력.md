# OMX 배송로봇 변경 이력

이 문서는 코드, 설정, 모델, launch, world, 인터페이스 또는 시험 결과가
바뀔 때 변경 근거와 검증 결과를 함께 기록하는 기준 문서다. 계획 상태는 각
계획서에서 관리하고, 실제로 적용된 변경은 이 문서에 날짜순으로 남긴다.

## 기록 기준

| 변경 대상 | 반드시 기록할 내용 |
|---|---|
| 코드·상태 머신 | 변경 이유, 영향받는 상태, 실패·복구 동작 |
| 파라미터 | 기존값, 변경값, 적용 범위가 실기인지 시뮬레이션인지 구분 |
| 모델·관측 계약 | 버전, checksum, 관측·행동 차원, 호환성 |
| launch·토픽 | 실행 진입점, 노드 소유권, 추가·삭제된 인터페이스 |
| 로봇·실험 world | 치수, 좌표, 질량, 충돌·마찰 조건 |
| 시험 결과 | 실행 조건, 최종 상태, 측정값, 실패 여부, 남은 한계 |

변경과 문서 갱신은 같은 작업에서 완료한다. 시뮬레이션 단일 성공은 반복
성공률이나 실기기 성공으로 확대 해석하지 않는다.

## 2026-07-17: RL Gazebo Pick-Place 물리 검증

### 범위

학습 정책 `arm_delivery_residual_v2`를 `omx_rl_control`에 적용하고,
Gazebo의 13 x 13 x 17 cm 타워와 6 x 5.5 x 5.5 cm 상자를 대상으로
`Pick -> Stay -> Place -> Stay` 전체 팔 동작을 검증했다. 베이스 이동과
UWB 배송은 포함하지 않은 arm-only 시험이다.

### 적용 변경

| 영역 | 변경 내용 | 근거 |
|---|---|---|
| 그리퍼 결과 | `allow_stall=true`인 Close에서만 `ABORTED + stalled`를 접촉 성공으로 허용 | Gazebo 그리퍼가 물체 접촉을 status 6과 `stalled=true`로 반환 |
| 파지 명령 | 정책 정규화의 close `-0.010 m`와 물리 명령 `gripper_grasp_position`을 분리 | 관측 계약을 유지하면서 5.5 cm 상자 폭에 맞춤 |
| Gazebo 파지 위치 | `0.0045 m` | `(0.055 - 0.004) / 2 - 0.021` 폭 기반 계산 |
| 릴리스 안정화 | 기본 `0.24 s`, Gazebo `0.75 s` | 학습 환경의 12 step settle을 재현하고 상자 끌림 방지 |
| 배송 목표 보정 | Gazebo EEF 목표 X를 `0.293 m`로 설정 | 파지된 상자 중심이 EEF 기준보다 약 2.3 cm 뒤에 위치 |
| 관절 상태 timeout | Gazebo에서만 `0.25 -> 1.0 s` | 렌더링 부하 중 일시적인 joint state 지연 허용 |
| trajectory 시간 | Gazebo에서만 `0.08 -> 0.25 s` | 지연 수신된 80 ms trajectory의 과거 종료 거부 방지 |

실기 기본값과 정책 관측 정규화 값은 변경하지 않았다. Gazebo 보정값은
`rl_gazebo.launch.py`의 노드 파라미터 override로 한정한다.

### 문제와 확인 과정

| 관찰 결과 | 원인 | 조치 |
|---|---|---|
| Close 직후 `FAULT` | 접촉 stall을 Action 실패로만 처리 | 허용된 접촉 stall 판정 추가 |
| 그리퍼 Close timeout | 완전 닫힘 목표가 5.5 cm 상자 폭과 불일치 | 폭 기반 파지 목표와 effort 적용 |
| 상태는 `COMPLETE`지만 상자가 바닥에 떨어짐 | Open 직후 팔이 복귀하며 상자를 끌고 감 | 릴리스 대기와 배송 X 오프셋 적용 |
| `joint3`가 waypoint에서 정체 | 80 ms trajectory가 수신 시점에 이미 종료됨 | Gazebo trajectory 시간을 250 ms로 확대 |

### 최종 결과

| 항목 | 결과 |
|---|---|
| 상태 전이 | `PICK_REACH -> PICK_TO_STAY -> WAIT_DELIVERY -> PLACE_REACH -> OPEN_RELEASE -> PLACE_TO_STAY -> COMPLETE` |
| 파지 유지 | 상자가 초기 위치에서 Stay 위치까지 EEF와 함께 이동 |
| 최종 상자 중심 | `(0.271851, 0.000003, 0.197500) m` |
| 타워 기준 중심 | `(0.270000, 0.000000, 0.197500) m` |
| 최종 판정 | 타워 상판 위 물리 배치 성공 |
| 자동 시험 | `19 passed, 1 skipped` |
| 빌드 | `colcon build --packages-select omx_rl_control` 성공 |

검증 세션은 `ROS_DOMAIN_ID=175`에서 실행했으며, 이는 다른 ROS 그래프와
분리하기 위한 시험값이다. 프로젝트의 고정 도메인 계약은 아니다.

## 2026-07-17: `mp_control` 대비 RL 시간 분석

### 비교 방법

RL은 최종 Gazebo 성공 로그의 상태 전이 시각을 사용했다.
`mp_control`은 동일 world 실측 로그가 없어 현재 통합 YAML과 C++ 상태
머신의 고정 대기 시간을 합산했다. 따라서 `mp_control` 값은 실측값이 아닌
낮게 잡은 구성상 최소값이다.

### 결과

| 항목 | 결과 |
|---|---:|
| RL Pick 명령 -> `WAIT_DELIVERY` | 8.639 s |
| RL Place 시작 -> `COMPLETE` | 9.268 s |
| RL 능동 팔 동작 합계 | 17.907 s |
| RL 명령 대기 포함 wall time | 19.971 s |
| `mp_control` 파지 후 동작 최소 | 24.600 s |
| RL 단축량 | 6.693 s, 27.2% |
| 속도비 | RL 1.37배 |

RL 정책은 CPU, PyTorch thread 1개 조건에서 예열 200회 후 5,000회
추론했다. 평균 0.124 ms, 중앙값 0.121 ms, p95 0.146 ms, p99
0.182 ms, 최대 0.276 ms였고 18 ms deadline 초과는 0회였다.

### 판정과 한계

- 현재 설정의 팔 임무 완료 시간은 RL이 우위하다.
- 제어 주기도 RL 50 Hz가 `mp_control` 20 Hz보다 2.5배 촘촘하다.
- 순수 연산량은 신경망이 없는 `mp_control`이 작지만, 측정된 RL 추론
  시간은 p99 기준으로 20 ms 주기의 1% 미만이다.
- 동일 조건의 `mp_control` 실행과 30회 반복 성공률은 아직 계측하지
  않았으므로 실기기 성공률·안정성까지 RL 우위로 판정하지 않는다.

상세 시간 근거와 후속 A/B 절차는
[OMX RL 제어 계획서](<./OMX RL 제어 계획서.md>)에 반영했다.

## 2026-07-17: PPO 런타임 CUDA 전환

Gazebo와 실기기에서 공통으로 사용하는 `omx_rl_control`의 기본 PPO 추론
장치를 CPU에서 NVIDIA CUDA GPU로 변경했다.

| 항목 | 변경 후 값 |
|---|---|
| 런타임 파라미터 | `policy_device: cuda` |
| GPU | NVIDIA GeForce RTX 4050 Laptop GPU |
| PyTorch | `2.9.1+cu130` |
| CUDA 인식 | CUDA 13.0, `torch.cuda.is_available() == True` |
| 적용 범위 | ROS 2 PPO 런타임과 Gazebo 비교 실행 |

학습 이력을 보존하기 위해 모델의 `training_config.yaml`과 완료된 MuJoCo
50회 비교 스크립트의 CPU 조건은 변경하지 않았다. Gazebo 물리 계산은
계속 CPU에서 수행되므로 GPU 전환은 PPO 추론 장치에만 적용된다.

### Gazebo 반복 시험 안정화

CUDA 전환 후 반복 파일럿에서 두 번째 파지 접촉이 action 결과를 반환하지
않고 4초 뒤 `gripper close timeout`으로 끝나는 현상을 확인했다. Gazebo
controller의 접촉 진동이 기본 stall 속도 임계값 `0.001 rad/s`보다 커서
정지로 판정되지 않는 것이 원인이었다.

| Gazebo 전용 파라미터 | 값 |
|---|---:|
| `stall_velocity_threshold` | `0.02 rad/s` |
| `stall_timeout` | `1.0 s` |

이 값은 `gazebo_controller_manager.yaml`에만 적용하며 실제 하드웨어
controller 설정에는 적용하지 않는다. 반복 실행기는 명령 중복을 막기 위해
`RESET`, `PICK`, `PLACE`를 회차마다 한 번만 발행하도록 수정했다.

정식 비교는 GUI client를 계측 경로에서 제외하고 Gazebo server가 world,
물리, 센서를 소유하는 구조로 실행한다. EEF 카메라의 서버 측 GPU 렌더링을
위해 실험 world에 `Sensors` system과 `ogre2` render engine을 명시하고,
server는 `--headless-rendering` 및 NVIDIA PRIME offload 환경에서 시작한다.
ODE 물리 계산은 CPU, PPO 추론과 카메라 렌더링은 NVIDIA GPU가 담당한다.

한 server에서 실패 후 `RESET`만 수행하면 다음 회차의 arm/gripper 상태가
오염되는 현상이 확인돼 정식 50회 시험은 회차마다 Gazebo server를 새로
시작하는 독립 trial 방식으로 변경했다. `gazebo_server_batch.py`가 동일
master seed의 샘플 순서를 유지하면서 server 시작, 1회 Pick-Place, CSV
저장, server 종료를 자동 수행한다.

반복 시작 과정에서 종료된 controller manager의 DDS discovery 정보가 다음
spawner에 남는 경우가 있어, batch runner는 trial과 startup retry마다 서로
다른 `ROS_DOMAIN_ID`를 할당한다. server 준비 실패는 제어 실패로 집계하지
않고 같은 seed를 최대 2회 재시도한다.

`ros2 launch` 부모가 먼저 종료된 뒤 같은 process group의 `ign gazebo`
child가 남는 경우도 확인했다. batch cleanup은 부모 종료 여부와 관계없이
process group 전체에 `SIGINT`, `SIGTERM`, 마지막 `SIGKILL`을 순서대로
적용해 다음 회차의 CPU/GPU 자원에 영향을 주지 않도록 변경했다.

## 2026-07-18: 실제 Stay 기준 PPO 재학습 및 v3 배포

기존 v2의 학습 Stay `[0, 0, 1.38, -1.38] rad`가 실제 bringup Stay와 달라
첫 정책 명령에서 큰 관절 오차가 생길 수 있었다. MuJoCo keyframe과 reset,
복귀 목표를 실제 자세 `[0.104311, 0.027612, -0.001534, -1.638291] rad`로
통일해 전체 curriculum을 다시 학습했다.

| 항목 | 결과 |
|---|---:|
| 정책 | `arm_delivery_residual_v3_robot_stay` |
| 누적 PPO step | 660,000 |
| 전체 타워 Pick | 98/100, 충돌 2 |
| 전체 타워 Place | 99/100, 충돌 1 |
| Pick/Place 혼합 | 99/100, 충돌 1 |
| 베이스 정지 오차 | 93/100, 충돌 7 |
| 높이·크기 오차 | 88/100, 충돌 12 |
| 전체 Sim2Real | 95/100, 충돌 5 |

metadata가 MuJoCo qpos 앞 4개를 팔 관절로 오인하던 문제를 수정해 관절
이름으로 Stay를 추출한다. 파지 상태 시작은 물체별 IK로 초기화하고,
pregrasp 높이는 끝단 충돌 회귀 시험에 따라 `22.5 mm`로 맞췄다. 런타임
IK 수렴 기준도 `0.01 mm`로 강화해 학습 해와 parity를 맞췄다.

기존 v2 artifact는 비교와 rollback 용도로 보존하고 기본 로더만 v3로
전환했다. 동일 seed에서 v3의 Sim2Real은 v2 `94%` 대비 `95%`였지만,
높이 조건은 v2 `91%` 대비 `88%`다. 따라서 중앙 저속 실기 시험부터
진행하며 양의 X 끝단과 높이 오차 구간은 안전 제한을 유지한다.

런타임은 외부 bringup이 실제 Stay 허용오차에 들어오기 전 `STAY_EMPTY`로
전환하지 않는다. 독립 Gazebo launch에서는 RL 노드가 Stay 정렬을 담당한다.
